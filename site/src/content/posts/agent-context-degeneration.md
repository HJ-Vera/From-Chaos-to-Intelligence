> 长上下文是存储，AI Agent 缺失的是信息流管理。

过去一年，长上下文模型给了很多人一种乐观的预期：既然模型可以接收 128K token，甚至 1M token，那么我们似乎就可以把用户对话、检索文档、工具调用结果、执行日志、中间推理过程以及 Agent 之间的通信全部保留下来。于是，一个很自然的假设出现了：上下文越长，Agent 的记忆越强；上下文越完整，系统就越可靠。

但在真实的复杂 Agent 系统里，我越来越经常看到相反的情况。上下文确实变长了，系统却未必变聪明。它会忽略早期的重要约束，重复已经失效的结论，引用不相关的材料，在多轮执行后偏离原始目标。更微妙的是，有些关键证据明明就在上下文里，却没有真正进入最终推理。最后系统输出了一份看起来合理、语言流畅、结构完整的答案，但如果顺着证据链往回查，会发现中间的依赖关系已经断裂。

我把这种现象称为 **Context Degeneration，也就是上下文退化**。

上下文退化让我意识到，长上下文解决的是“信息能不能被放进去”的问题，而不是“信息能不能被正确组织、更新、追踪和使用”的问题。LLM 本身并不负责管理信息，它更像是在消费信息。真正决定系统可靠性的，是外部系统如何决定保留什么、遗忘什么、更新什么、压缩什么，以及哪些信息应该进入下一步推理。

所以，这篇文章想讨论的不是“长上下文有没有用”。长上下文当然有用，RAG 当然有用，多 Agent 也当然有用。但它们共同绕不开一个更底层的问题：**AI Agent 的核心并不是更长的上下文，而是结构化的信息流管理。**

## 一、信息在上下文里，不等于信息进入了推理

很多 Agent 系统默认相信一个假设：只要信息还在上下文里，模型总会在需要的时候找到它。这个假设在简单任务里通常问题不大，但在复杂任务里并不可靠。因为“信息存在于上下文”和“信息被模型稳定使用”之间，并没有必然关系。

我们至少需要区分三个概念：

| 概念 | 含义 |
| --- | --- |
| 名义上下文 | 模型理论支持的上下文长度 |
| 有效上下文 | 模型在当前任务中能够稳定利用的信息范围 |
| 信息密度 | 单位 token 中真正支持当前决策的信息比例 |

长上下文主要提升的是第一项，也就是模型理论上可以接收的信息容量。但复杂 Agent 真正依赖的，往往是后两项：有效上下文和信息密度。一个证据即使存在于上下文里，也可能被大量低价值文本稀释，可能和当前子任务的关系不够明显，可能被后续错误结论覆盖，也可能因为距离当前生成位置太远而没有被稳定利用。

因此，真正重要的问题不是“信息有没有被保存”，而是这条信息现在是否仍然有效、它支持哪个判断、它是否应该进入下一步推理，以及它是否需要被更新或遗忘。到了这一步，问题已经不再是上下文长度问题，而是状态管理问题。

这也是我理解 Agent 工程的第一层判断：**长上下文是存储，记忆是状态。** 顺着这个判断继续往下推，还可以得到两个更具体的命题：检索负责找到信息，记忆负责维护状态；LLM 消费信息，系统管理信息。

## 二、长上下文是仓库，不是工作记忆

把长上下文当成记忆，是很多 Agent 系统一开始最容易犯的错误。原因也很直观：既然模型能接收更多历史，那它似乎就“记得”更多东西。但人的工作记忆并不是这样运作的。真正有用的记忆不是记住所有东西，而是知道哪些信息值得保留，哪些结论已经被新证据覆盖，哪些历史内容已经过期，以及当前任务到底推进到了什么状态。

从这个角度看，长上下文更像仓库，而不是工作记忆。仓库关心的是“有没有保存”，工作记忆关心的是“现在能不能用”。仓库可以不断堆放材料，但工作记忆必须服务于当前行动。一个 Agent 如果只是把所有历史材料都放进上下文，它拥有的只是更大的存储空间，而不是更好的记忆能力。

从模型机制上看，这个区别也很重要。长上下文能力很大程度上依赖注意力计算中的 KV 缓存：模型在生成时会把已经处理过的 token 表示缓存下来，以便后续 token 继续访问这些历史信息。换句话说，KV 缓存让模型能够“保留可被访问的历史表示”，但这并不等于模型真的理解了哪些信息更重要，也不等于它会在正确的推理步骤中调用正确的信息。它提供的是上下文存储和访问条件，而不是任务状态管理能力。

这也是为什么长上下文模型仍然会遇到所谓的 `lost in the middle` 问题。相关研究观察到，模型对上下文不同位置的信息利用并不均匀，开头和结尾的信息往往更容易影响输出，而夹在中间的内容即使存在，也可能被弱化。在 Agent 系统里，这个问题会被进一步放大：工具结果、检索文档、历史计划和中间总结不断堆叠后，真正关键的约束可能并没有消失，却被埋在上下文中间，变成“看得见但用不上”的信息。

真正的 Agent 记忆，应该是一组可以读取、更新、压缩和审查的状态，而不是一段越滚越长的聊天记录。状态的价值在于它能告诉系统：当前目标是什么，哪些约束仍然有效，哪些证据支撑了当前判断，哪些假设还没有验证，哪些内容已经不应该继续影响后续推理。

这也是为什么单纯扩大上下文窗口，并不会自动带来更可靠的推理。复杂任务需要的不是材料全集，而是当前任务的可用状态。

## 三、RAG 的问题不只是检索

RAG 经常被理解为“检索相关资料，然后交给模型生成答案”。在简单问答里，这样做通常够用：用户问一个事实，系统找到相关段落，模型基于段落回答。但复杂任务不是这样。复杂任务需要的不只是相关材料，而是经过组织的证据状态。

例如，在一个复杂分析任务中，系统需要知道哪些材料是事实，哪些材料只是观点；哪些证据互相支持，哪些证据互相冲突；哪些证据对应当前问题，哪些证据只适合后续阶段使用；哪些结论依赖尚未验证的前提。如果检索结果只是被拼接进提示词，模型就必须在同一次生成中完成阅读、筛选、归类、推理和表达。

这就是很多 RAG 系统不稳定的原因。它完成了信息获取，但没有完成信息组织。检索只是让信息进入系统，真正决定系统质量的，是这些信息进入之后如何被清洗、归类、结构化、路由，并最终转化成可推理的状态。

所以，更合理的 RAG 流程不应该停留在：

```text
检索 -> 拼接 -> 生成
```

而应该更接近：

```text
检索 -> 清洗 -> 归类 -> 结构化 -> 路由 -> 推理
```

这里的关键变化是，检索结果不再被当作一堆等待模型自行理解的文本，而是被转化成系统可以管理的中间状态。只有这样，RAG 才能从“找资料”走向“支持决策”。

## 四、上下文退化：Agent 越跑越乱的六种模式

多轮 Agent 系统最危险的地方在于，它往往不是立刻失败，而是逐渐变差。前几轮看起来很正常，随着工具调用增加、检索材料增加、中间总结增加，系统开始变得含糊、重复、漂移，最后输出一个看似完整但证据混乱的答案。

我认为，上下文退化至少有六种常见模式：

| 退化模式 | 表现 | 本质问题 |
| --- | --- | --- |
| 注意力稀释 | 关键证据被大量低价值文本淹没 | 信息密度下降 |
| 状态漂移 | 当前任务状态逐渐偏离原始目标 | 目标没有被持续维护 |
| 错误传播 | 早期错误被后续步骤继续引用和放大 | 中间状态缺少校验 |
| 过期上下文 | 已失效的信息仍然影响判断 | 系统不知道该遗忘什么 |
| 证据丢失 | 证据存在于上下文中，但没有进入最终结论 | 证据没有被路由到推理步骤 |
| 角色混乱 | 多个 Agent 的职责、输出和判断边界互相污染 | 状态边界不清晰 |

这些模式解释了很多 Agent 系统的失败。模型明明检索到了正确资料，却在最终回答里使用了另一个更泛化的说法；某个早期计划已经不适用，但后续 Agent 仍然沿着旧计划继续执行；一个中间步骤里的猜测，被后续步骤当成事实继续传播；多个 Agent 都在“补充分析”，却没有任何一个 Agent 真正负责维护系统当前状态。

这也说明，多 Agent 并不天然可靠。如果多个 Agent 只是轮流往同一个上下文里追加文本，那么系统复杂度增加了，但信息结构没有变清楚。这样的多 Agent 系统，本质上只是把一个混乱的长上下文拆成了几个互相影响的长上下文。

Agent 不是越多越好。关键是每个 Agent 是否处理清楚的状态，是否只接收当前阶段需要的信息，是否把结果转化成下游可以使用的结构。

## 五、Agent 缺失的一层：信息流管理

一个可靠的 Agent 系统，不应该把上下文当成一个大文本框，而应该把它看成一条信息流。系统需要持续回答：什么信息进入系统，什么信息被保留，什么信息被压缩，什么信息被更新，什么信息被遗忘，什么信息进入下一步推理，以及什么信息只作为日志保存、不参与决策。

这就是信息流管理。它关心的不是提示词怎么写得更漂亮，而是信息如何在系统中移动。

一个更可靠的 Agent 流程大致会是这样：

```mermaid
flowchart LR
    A[用户问题] --> B[问题建模]
    B --> C[检索资料]
    C --> D[证据整理]
    D --> E[状态更新]
    E --> F[任务推理]
    F --> G[质量检查]
    G --> H[最终回答]
```

在这个流程里，每一步都不是简单生成一段文字，而是在更新系统状态。问题建模不是复述用户需求，而是把需求转化成目标、约束和判断标准；证据整理不是摘要文档，而是判断哪些信息可以支持决策；状态更新不是压缩上下文，而是保留下一步推理真正需要的依赖关系；质量检查也不是让模型评价“答案看起来不错”，而是检查状态是否完整、证据是否匹配、结论是否越界。

也就是说，Agent 的可靠性不只来自模型能力，还来自系统对信息流的治理能力。

## 六、结构化状态比长文本更重要

如果说长上下文解决的是“把信息放进去”，那么结构化状态解决的是“让信息能被使用”。一个 Agent 不应该只保存一段笼统的历史，比如“用户问了一个复杂问题，系统检索了五篇资料，模型生成了若干分析，工具返回了一些结果，然后模型继续推理”。这样的历史虽然完整，却很难直接支持下一步决策。

更合理的方式，是让系统维护不同类型的状态：

| 状态类型 | 需要记录的内容 |
| --- | --- |
| 任务状态 | 目标是什么，当前进行到哪一步 |
| 约束状态 | 哪些条件必须满足，哪些条件不能违反 |
| 证据状态 | 每个结论对应哪些来源 |
| 风险状态 | 哪些判断存在不确定性 |
| 决策状态 | 哪些方案被保留，哪些方案被排除 |

这样做的价值不是让系统看起来更复杂，而是降低模型每一轮的认知负担。模型不需要在一大段历史文本中重新寻找重点，只需要读取当前阶段需要的状态，并基于这些状态完成下一步推理。

这也是 Context Engineering 的核心：不是把更多内容塞进上下文，而是设计上下文的结构。更具体地说，是设计信息如何被表示、传递、压缩、更新和遗忘。

## 七、CGAP：我第一次意识到，Agent 的问题不是检索，而是状态

我第一次清楚意识到这件事，是在做毕业论文的时候。我的研究方向是城市规划中的案例迁移。规划师经常参考其他城市的成功案例，但一个城市成功，不代表另一个城市可以直接复制，因为制度环境、土地权属、财政能力、空间结构、公众接受度和利益相关者关系都可能不同。

最开始，我把它当作一个检索问题：找到相似案例，然后交给模型生成建议。但很快我发现，问题根本不在“有没有找到相关案例”，而在于这个案例成功所依赖的前提条件，在本地是否存在。

如果把国际案例、本地背景、政策文件、空间条件、公众争议和实施风险全部塞进提示词，模型当然能写出一份逻辑通顺、语言漂亮、看起来很专业的建议。但它经常回答不了真正关键的问题：这个案例为什么成功，本地缺少什么条件，哪些条件可以适配，哪些条件根本无法迁移，以及哪些风险会在实施阶段爆发。

我后来意识到，这不是 Agent 有没有找到信息的问题，而是 Agent 有没有维护正确状态的问题。所以，我设计的 CGAP 虽然形式上是一个多 Agent 工作流，但本质上更像一个状态转换系统。每一个 Agent 都不应该只是“生成一段分析”，而应该负责更新系统状态。

| 阶段 | 状态变化 |
| --- | --- |
| Context Modeling | 把本地问题转化成空间、制度、经济和社会约束 |
| Case Retrieval | 找到结构上相关的案例，而不只是文本相似的案例 |
| Gap Analysis | 比较外部案例的成功前提和本地条件之间的差距 |
| Review | 检查中间状态是否完整、一致、可继续推理 |

这套流程的重点不是“多用了几个 Agent”，而是每一步都在控制信息如何进入下一步。本地背景不会作为一大段文本被动传递，而是被转化成约束状态；外部案例不会作为网页片段直接进入最终回答，而是被重建成前提、策略和结果；差距分析不会依赖模型临场发挥，而是要求模型明确比较“案例成功条件”和“本地现实条件”之间的缺口。

从这个角度看，CGAP 不是一个孤立案例，而是这篇文章的证据。它证明了一个更通用的判断：复杂任务里的 Agent 不应该只是检索和生成，而应该持续维护状态。甚至可以说，所谓 Agent 不应该只是会说话的角色，它应该是带有 LLM 接口的状态机。这个说法听起来有点激进，但我越来越觉得它接近复杂 Agent 系统的本质。

## 八、可靠 AI 系统，本质上是可靠的信息流系统

现在很多 Agent 讨论会集中在模型、工具、长上下文、插件和框架上。这些当然重要，但它们都没有替代一个更底层的问题：在一个可信的 AI 系统里，信息应该如何流动？

信息如何进入系统，如何被理解，如何被压缩，如何被传播，如何被更新，如何被遗忘，最后如何形成可靠决策。这可能是下一阶段 Agent Engineering 最重要的问题之一。

长上下文解决的是“放得下”，RAG 解决的是“找得到”，工具调用解决的是“做得到”，多 Agent 解决的是“能分工”。但复杂任务还需要解决一个更底层的问题：管得住。

如果没有信息流管理，这些能力仍然可能堆成一个不稳定的系统。LLM 本身并不管理信息，它只是消费信息。真正负责组织信息的，应该是系统。

因此，可靠 Agent 的关键不是记住所有东西，而是知道什么应该保留，什么应该遗忘，什么应该更新，什么应该进入下一步推理。长上下文是存储，记忆是状态，而 AI Agent 真正缺失的那一层，或许正是信息流管理。
